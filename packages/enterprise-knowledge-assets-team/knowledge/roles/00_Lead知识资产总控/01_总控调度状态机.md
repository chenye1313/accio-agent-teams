# 01 总控调度状态机

版本：v0.1  
适用角色：`00_Lead知识库基座总控`

## 1. 核心职责

知识库基座总控不替代执行角色，而是负责：

1. 判断企业知识库基座当前处于哪个阶段。
2. 确认是否可以进入下一阶段。
3. 分发任务给资料接收与托管师、知识库构建师、知识库质量与分发官。
4. 审查交接物是否通过质量门。
5. 将可用知识分发给六组，并定义回流要求。

## 2. 标准调用顺序

```text
00_Lead 知识库基座总控
  -> 01_资料接收与托管师
  -> 02_知识库构建师
  -> 03_知识库质量与分发官
  -> 00_Lead 最终放行
  -> 六组数字化经营团队
  -> 新问答 / 新案例 / 新证据回流
```

## 3. 状态机

```yaml
knowledge_asset_state_machine:
  context_ready:
    next: "source_intake"
    block_if:
      - "business context missing"
      - "target output unclear"
      - "privacy requirement unclear"

  source_intake_ready:
    next: "library_strategy"
    block_if:
      - "raw material list missing"
      - "source owner unknown"
      - "privacy risk unmarked"

  library_strategy_ready:
    next: "asset_building"
    block_if:
      - "four library scope unclear"
      - "method treated as external fifth library"
      - "field requirements missing"

  asset_building_ready:
    next: "quality_gate"
    block_if:
      - "library candidates missing source id"
      - "inference not marked"
      - "case or qa lacks core fields"

  quality_gate_ready:
    next: "six_group_distribution"
    block_if:
      - "privacy gate not run"
      - "evidence level missing"
      - "unsupported claim included"

  distribution_ready:
    next: "feedback_loop"
    block_if:
      - "usage scenario missing"
      - "downstream group unclear"
      - "feedback requirement missing"

  feedback_loop_ready:
    next: "asset_center_iteration"
    block_if:
      - "new qa/case/evidence not classified"
      - "update owner missing"
```

## 4. 阶段判定

| 状态 | 含义 | 总控动作 |
|---|---|---|
| `context_ready` | 已知道企业目标和建设范围 | 分派资料接收任务 |
| `source_intake_ready` | 资料已登记、脱敏、编号 | 制定四库建设策略 |
| `library_strategy_ready` | 四库范围和字段已定 | 分派知识构建任务 |
| `asset_building_ready` | 四库候选条目已形成 | 进入证据和隐私质检 |
| `quality_gate_ready` | 质量门通过或缺口明确 | 生成六组分发包 |
| `distribution_ready` | 分发包可使用 | 定义回流要求 |
| `blocked` | 关键条件不足 | 输出返工角色和补充要求 |
