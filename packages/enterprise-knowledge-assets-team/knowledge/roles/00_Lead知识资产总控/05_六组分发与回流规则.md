# 05 六组分发与回流规则

版本：v0.1  
适用角色：`00_Lead知识库基座总控`

## 1. 分发原则

企业知识库基座不把完整四库一股脑丢给六组，而是按任务打包。

每个分发包必须包含：

```yaml
distribution_package:
  target_group: ""
  task_context: ""
  knowledge_items: []
  allowed_usage: []
  forbidden_usage: []
  evidence_gaps: []
  feedback_required: []
```

## 2. 六组默认分发包

| 六组 | 默认知识包 | 使用场景 | 回流要求 |
|---|---|---|---|
| 经营定位组 | 企业优势包、客户问题包、案例证据包、经营缺口包 | 判断服务谁、凭什么赢、先做哪盘生意 | 新定位、证据缺口、需要补访谈的问题 |
| 选品定品组 | 产品线包、SKU 表、客户需求包、案例中的产品反馈 | 判断主推品、机会品、淘汰品 | 产品表现、客户需求、SKU 取舍原因 |
| 内容表达与设计组 | 企业故事包、产品卖点包、案例信任包、FAQ 内容包 | 店铺首页、详情页、主图、短视频脚本 | 新素材、内容效果、表达缺口 |
| 推流运营组 | 产品标签包、关键词线索包、客户问题包、案例场景包 | 关键词、投放素材、流量标签、运营复盘 | 高质量关键词、流量反馈、素材测试结果 |
| 商机转化组 | 异议处理包、报价问答包、案例证明包、产品交付包 | 询盘回复、报价解释、跟进、谈单 | 新问题、新异议、成交和丢单原因 |
| 私域沉淀组 | 客户标签包、复购线索包、客户反馈包、新案例回收表 | 老客分层、复购触达、客户资产沉淀 | 新案例、复购反馈、客户标签 |

## 3. 使用边界

```yaml
usage_boundary:
  enterprise_info:
    allowed: "信任证明、公司能力、经营定位、内容素材"
    forbidden: "无证据夸大为行业第一"
  product_info:
    allowed: "选品、详情页、报价解释、关键词标签"
    forbidden: "把未验证卖点写成确定承诺"
  customer_qa:
    allowed: "成交回复、FAQ、内容选题、异议处理"
    forbidden: "脱离买家语境做通用回复"
  customer_case:
    allowed: "信任资产、销售证明、复盘学习"
    forbidden: "未授权公开客户名或成交价"
```

## 4. 回流协议

六组每次使用企业知识库基座知识后，至少回流三类信息：

```yaml
feedback_loop:
  new_question:
    source_group: ""
    buyer_context: ""
    question: ""
    proposed_answer: ""
  new_case:
    source_group: ""
    buyer_type: ""
    problem: ""
    solution: ""
    result: ""
  new_evidence:
    source_group: ""
    related_library_item: ""
    evidence_type: ""
    impact: "confirm/update/reject"
```

## 5. 返工触发

必须回到企业知识库基座更新：

- 六组发现客户持续提出新异议。
- 选品或运营数据推翻原产品判断。
- 成交组发现旧回答转化差。
- 私域组拿到复购或售后反馈。
- 内容表达组发现案例不够公开或证据不够强。

## 6. 禁止

- 禁止没有回流要求的分发。
- 禁止六组修改四库原始条目但不回写。
- 禁止把一次失败直接当成知识结论，必须标注为观察项或待验证。
