# 脚本优先执行层

## 核心判断

企业知识库基座不是把所有资料都交给 AI 逐条处理。AI 只负责语义判断、抽象、取舍、冲突裁决和最终表达。

凡是满足以下任一条件的步骤，必须优先脚本化：

- 输入输出规则稳定。
- 可以用字段、ID、页码、目录、引用关系验证。
- 需要重复执行很多次。
- 失败后需要可复现错误。
- 不需要价值判断，只需要准确搬运、切分、计数、校验或格式化。

## 脚本负责什么

| 环节 | 脚本责任 | AI 责任 |
|---|---|---|
| 资料接收 | 文件清单、front-matter、页码、字数、可读性、重复文件 | 判断资料价值和缺口 |
| 文档解析 | PDF 转文本、分页、章节切分、目录树、来源索引 | 判断哪些章节高相关 |
| 批次拆分 | 生成微任务表、限制单次任务体量、检查空目录 | 判断优先级和预算 |
| 条目校验 | ID 唯一、引用闭合、source 字段、quote 长度、schema 字段 | 判断条目是否有用、是否过度抽象 |
| 质量门 | 机械错误、缺字段、悬挂引用、重复 ID、格式超限 | 证据是否支撑结论、是否口吻迁移、是否可分发 |
| 分发包 | 打包目录、生成索引、统计条目数量 | 决定哪些条目喂给哪一组 |

## AI 必须负责什么

- 从原文中识别真正的原理、框架、案例、术语、反模式。
- 把材料变成可迁移知识，而不是摘抄句子。
- 判断同一案例是否应该合并、降级或剔除。
- 判断知识条目能否服务用户的实际内容场景。
- 判断口吻迁移、AI-Slop、过度模板化等语义风险。
- 对冲突证据做裁决，并标注不确定性。

## 微任务粒度门

大型条目抽取不得一次性派给单个 Agent。

默认上限：

```yaml
one_ai_task:
  raw_sections: 1
  libraries: 1
  max_new_items: 5
  must_run_script_check_after_write: true
```

触发拆分：

- 单次任务超过 1 个章节文件。
- 单次任务要求覆盖 2 个以上知识库。
- 单次任务新增条目超过 5 条。
- Agent 出现一次笼统报错。
- 输出目录创建了但条目为空。

出现连续两次笼统报错时，禁止继续盲试，必须先脚本检查目录和已有产物。

## 团队脚本

脚本路径：

```bash
agent-ready/Teams/企业知识库基座Team/scripts/knowledge_pipeline_tools.py
```

常用命令：

```bash
python3 agent-ready/Teams/企业知识库基座Team/scripts/knowledge_pipeline_tools.py inventory --kb-root output/读书知识库/内容红利
python3 agent-ready/Teams/企业知识库基座Team/scripts/knowledge_pipeline_tools.py check --kb-root output/读书知识库/内容红利
python3 agent-ready/Teams/企业知识库基座Team/scripts/knowledge_pipeline_tools.py microtasks --kb-root output/读书知识库/内容红利 --batch batch_1 --sections 00_preface.md --libraries principles,anti_patterns --max-items 5 --write
```

## 总控放行规则

总控不得只凭 Agent 口头汇报判断完成。至少需要以下证据之一：

- 脚本 `check` 输出 `error_count=0`。
- 条目文件中存在新增 ID，且 ID 无重复。
- 引用闭合检查无 missing_ref。
- 质量门报告列出具体文件路径、条目 ID 和问题类型。

没有脚本证据的“已完成”，只能标为 `claimed_complete`，不能标为 `verified_complete`。
