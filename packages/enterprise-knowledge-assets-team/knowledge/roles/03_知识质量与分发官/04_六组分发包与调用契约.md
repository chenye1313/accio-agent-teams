# 04 六组分发包与调用契约

版本：v0.1  
适用角色：`03_知识质量与分发官`

## 1. 分发原则

企业知识库基座不把四库整体交给六组，而是按任务打包。

每个分发包必须包含：

```yaml
distribution_package:
  package_id: ""
  target_group: ""
  task_context: ""
  source_libraries: []
  knowledge_items: []
  allowed_usage: []
  forbidden_usage: []
  evidence_gaps: []
  privacy_boundary: ""
  feedback_required: []
```

## 2. 六组默认调用关系

| 六组 | 优先调用 | 默认分发包 |
|---|---|---|
| 经营定位组 | 企业信息库、客户案例库、客户问答库 | 企业优势包、客户问题包、案例证据包、经营缺口包 |
| 选品定品组 | 产品信息库、客户案例库、客户问答库 | 产品线包、SKU 包、客户需求包、产品反馈包 |
| 内容表达与设计组 | 企业信息库、产品信息库、客户案例库、客户问答库 | 企业故事包、产品卖点包、案例信任包、FAQ 内容包 |
| 推流运营组 | 产品信息库、客户问答库、客户案例标签 | 产品标签包、关键词线索包、客户问题包、场景包 |
| 商机转化组 | 客户问答库、客户案例库、产品信息库 | 异议处理包、报价问答包、案例证明包、交付解释包 |
| 私域沉淀组 | 客户案例库、客户问答库、客户标签 | 客户标签包、复购线索包、客户反馈包、新案例回收表 |

## 3. 调用边界

```yaml
usage_contract:
  enterprise_info:
    allowed: ["经营定位", "公司能力说明", "信任证明", "内容素材"]
    forbidden: ["无证据夸大", "伪造行业排名", "把待验证经验写成事实"]
  product_info:
    allowed: ["选品", "详情页", "报价解释", "关键词标签", "交付说明"]
    forbidden: ["承诺未验证能力", "省略限制条件", "夸大参数"]
  customer_qa:
    allowed: ["询盘回复", "FAQ", "内容选题", "异议处理"]
    forbidden: ["脱离买家语境", "把内部经验直接外发", "覆盖隐私信息"]
  customer_case:
    allowed: ["信任资产", "销售证明", "复盘学习", "案例内容"]
    forbidden: ["公开未授权客户名", "公开精确成交价", "编造成交结果"]
```

## 4. 分发包质量门

分发前必须检查：

- 只分发 `approve` 或 `approve_with_boundary` 条目。
- `needs_validation` 只能作为内部观察，不进入外部内容。
- `internal_only` 只能分发给内部决策、培训或复盘。
- 每个条目必须带 `item_id`、`source_id`、`evidence_level`、`permission_status`。
- 每个包必须写清楚禁止用法。
- 每个包必须有回流要求。

## 5. 分发输出

```yaml
six_group_distribution_plan:
  packages:
    - package_id: ""
      target_group: ""
      task_context: ""
      items:
        - item_id: ""
          source_library: ""
          evidence_level: ""
          permission_status: ""
          usage_note: ""
      allowed_usage: []
      forbidden_usage: []
      feedback_required: []
      owner: "03_知识质量与分发官"
      approved_by: "00_Lead知识资产总控"
```

## 6. 返工触发

必须暂缓分发：

- 条目质量低于 16 分。
- 权限未知但目标用途是公开内容。
- 六组任务与知识来源不匹配。
- 案例证据不足却被用于销售证明。
- 产品信息无法支撑承诺。
