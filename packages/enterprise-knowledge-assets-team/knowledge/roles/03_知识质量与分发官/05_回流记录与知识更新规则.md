# 05 回流记录与知识更新规则

版本：v0.1  
适用角色：`03_知识质量与分发官`

## 1. 回流目的

企业知识库基座不是一次性建库，而是周期性更新企业知识资产。六组使用知识后必须回流：

```text
客户又问了什么？
哪些说法有效？
哪些判断被数据推翻？
有没有新案例、新反馈、新证据？
```

## 2. 回流类型

```yaml
feedback_loop:
  new_question:
    source_group: ""
    buyer_context: ""
    question: ""
    proposed_answer: ""
    related_item_id: ""
  new_case:
    source_group: ""
    buyer_type: ""
    problem: ""
    solution: ""
    result: ""
    permission_status: ""
  new_evidence:
    source_group: ""
    related_library_item: ""
    evidence_type: "backend | chat | order | customer_feedback | manual_review"
    impact: "confirm | update | reject | split"
  usage_result:
    source_group: ""
    package_id: ""
    task_context: ""
    result_summary: ""
    conversion_signal: ""
```

## 3. 更新动作

| 回流内容 | 更新动作 |
|---|---|
| 新问题 | 进入客户问答库候选 |
| 新异议 | 更新 `objection_type` 和推荐用法 |
| 新成交案例 | 进入客户案例库候选 |
| 新丢单复盘 | 进入案例或 QA 的反例标签 |
| 新后台数据 | 提升或降低证据等级 |
| 新产品反馈 | 更新产品信息库场景、卖点或限制 |
| 内容/运营效果 | 更新六组分发规则 |

## 4. 证据升级与降级

```yaml
evidence_update_rule:
  upgrade:
    C_to_B: "口述经验被聊天、录音、文档或客户反馈验证"
    B_to_A: "文档或案例被后台数据、订单或成交记录验证"
  downgrade:
    A_to_B: "数据口径变化或记录不完整"
    B_to_C: "来源无法复核或权限被收回"
    any_to_D: "发现编造、冲突无法解释、隐私违规"
```

## 5. 更新请求

```yaml
knowledge_update_request:
  request_id: ""
  triggered_by_group: ""
  related_package_id: ""
  related_item_id: ""
  update_type: "add | revise | merge | split | downgrade | archive"
  reason: ""
  evidence: []
  return_to_role: "01_资料接收与托管师 | 02_知识资产构建师 | 00_Lead知识资产总控"
  priority: "high | medium | low"
```

## 6. 禁止

- 禁止六组直接改正式库但不回写。
- 禁止把单次使用效果当作确定结论。
- 禁止用未脱敏客户反馈补公开案例。
- 禁止只记录成功案例，不记录丢单和失败反馈。
