# 02 资料拆分覆盖清单

版本：v0.1  
适用角色：`02_知识资产构建师`

## 1. 总原则

资料不是按文件名拆，而是按“能形成什么知识资产”拆。

每份资料至少要判断：

```yaml
coverage_decision:
  source_asset_id: ""
  material_type: ""
target_library:
    - "enterprise_info"
    - "product_info"
    - "customer_qa"
    - "customer_case"
    - "evidence_layer"
    - "content_application_route"
  extractable_fields: []
  missing_fields: []
  inference_needed: true
```

## 2. 公司介绍 / 官网 / PPT

必须尝试覆盖：

- `company_profile`：成立时间、地区、规模、主营业务。
- `capabilities`：工厂、研发、设计、打样、生产、交付、服务能力。
- `certifications`：认证、证书、检测、专利、平台背书。
- `service_scope`：服务市场、客户类型、订单类型、行业场景。
- `advantages`：优势主张，但必须标注证据。
- `team`：团队、经验、关键岗位。
- `development_stage`：发展阶段、主营阶段、平台阶段。
- `evidence`：来源文件、页面、证书、访谈。
- `brand_claims`：品牌主张、对外品宣句、价值观、质量承诺。
- `public_or_internal_boundary`：哪些可公开，哪些只供内部理解。

## 3. 产品目录 / SKU 表 / 报价单 / 详情页

必须尝试覆盖：

- `product_line`：产品线、品类、系列。
- `sku`：型号、规格、款式、变体。
- `features`：功能、材质、参数、卖点。
- `use_cases`：应用场景、使用人群、采购场景。
- `customization`：尺寸、颜色、LOGO、包装、工艺。
- `delivery`：MOQ、打样周期、生产周期、物流、售后。
- `proof`：报价单、详情页、样品、订单、后台表现。
- `buyer_fit`：适合哪些买家，不确定则标记推断。

## 4. 询盘 / 聊天 / 邮件 / OKKI / 客服记录

必须尝试覆盖：

- `question`：客户原始问题或归一问题。
- `buyer_context`：国家、买家类型、采购阶段、使用场景。
- `answer`：已有回答或推荐回答候选。
- `answer_structure`：核心观点、事实依据、案例、步骤、边界、可复用公式。
- `application_scenarios`：适合落地的业务场景，如询盘回复、报价解释、详情页 FAQ、销售培训、内容选题。
- `objection_type`：价格、MOQ、交期、质量、认证、售后、定制、付款。
- `tags`：主题、产品、客户类型、异议、内容方向等标签。
- `question_keywords`：客户可能搜索、提问或运营可投放的关键词。
- `source`：聊天、询盘、邮件、录音、人工补充。
- `recommended_usage`：成交回复、FAQ、详情页、销售培训、内容选题。
- `related_product`：关联 SKU 或产品线。

## 5. 客户录音 / 访谈 / 老板口述 / 销售复盘

必须尝试覆盖：

- `voice_context`：语音来源、沟通场景、参与角色、时间点。
- `customer_question`：客户问了什么。
- `customer_context`：客户是谁、什么场景、什么阶段。
- `answer_fragment`：销售、老板或客服当时如何回答。
- `scenario_fragment`：这个问答可以迁移到什么业务场景。
- `tag_fragment`：可沉淀的主题标签、问题标签、产品标签和关键词。
- `brand_claim`：是否出现企业故事、品牌主张、工厂实力、质量标准。
- `product_expression`：是否出现产品卖点、参数解释、应用场景、定制能力。
- `promotion_angle`：是否可转成对外品宣、营销推广或社媒内容。
- `case_signal`：是否出现案例线索。
- `experience_signal`：是否出现方法经验。
- `sales_tactic`：销售动作、谈判动作、跟进动作。
- `decision_judgment`：老板或销售的判断标准。
- `needs_validation`：哪些内容只是口述经验。

语音和访谈不能只输出转写稿，必须同时输出结构化内容资产：

```yaml
voice_content_asset:
  source_id: ""
  source_type: "boss_interview/customer_call/sales_review/trade_show/offline_visit"
  context:
    scene: ""
    product_line: ""
    buyer_stage: ""
  extracted_assets:
    brand_claims: []
    company_capabilities: []
    product_selling_points: []
    buyer_questions: []
    answers: []
    case_signals: []
    proof_points: []
    content_angles: []
    risk_or_internal_only: []
  output_to_libraries:
    enterprise_info: []
    product_info: []
    customer_qa: []
    customer_case: []
    content_application_route: []
```

## 6. 成交 / 交付 / 客户反馈

必须尝试覆盖：

- `buyer_type`：客户类型、国家、行业、角色。
- `problem`：客户问题或采购挑战。
- `conversation_summary`：沟通过程摘要。
- `solution`：产品、服务、方案或跟进动作。
- `result`：成交、样品、复购、交付、客户反馈。
- `reusable_method`：可复用经验，内部字段。
- `public_permission`：能否公开。
- `evidence_level`：初步证据强弱，最终由质检官确认。

## 7. 后台 / OKKI / 询盘 / 成交数据

必须尝试覆盖：

- `product_performance`：产品曝光、点击、询盘、成交。
- `buyer_signal`：国家、行业、采购阶段、客户质量。
- `qa_frequency`：高频问题。
- `case_evidence`：是否能补强案例结果。
- `proof`：作为四库证据层，不单独变成对外第五库。

## 8. 品牌内容 / 对外品宣 / 社媒独立站素材

必须尝试覆盖：

- `brand_story`：企业来历、老板故事、工厂故事。
- `company_positioning`：公司定位、服务客户、核心差异。
- `capability_claims`：产能、设计、研发、交付、服务能力。
- `quality_principles`：质量标准、不可妥协点、检测流程。
- `promotion_angle`：适合对外传播的角度。
- `usable_channels`：阿里国际站公司页、LinkedIn、独立站、销售手册、邮件。
- `proof_points`：证书、案例、数据、客户反馈。
- `do_not_claim`：不能承诺、不能公开、需要验证的内容。

输出建议：

```yaml
brand_content_asset:
  brand_story:
  founder_or_factory_story:
  company_positioning:
  capability_claims:
  target_buyer:
  proof_points:
  usable_channels:
  do_not_claim:
```

## 9. 内容应用路由

所有四库条目都要尽量判断能不能进入内容应用：

```yaml
content_application_route:
  source_library: ""
  content_layer: "enterprise_strength/brand_promotion/product_expression/buyer_qa/trust_evidence/scenario_solution"
  source_format: "document/voice_transcript/offline_interview/chat_record/backend_data"
  application_target:
    - "alibaba_knowledge_base"
    - "structured_product_detail"
    - "faq_pack"
    - "inquiry_followup"
    - "brand_promotion"
    - "marketing_content"
    - "social_independent_site"
  required_fields: []
  quality_gate: []
```

## 10. 缺失字段输出

```yaml
missing_coverage:
  asset_id: ""
  target_library: ""
  missing_fields: []
  suggested_fix:
    - "补资料"
    - "补访谈"
    - "补后台数据"
    - "补客户授权"
```
