# 04 企业与产品信息抽取规则

版本：v0.1  
适用角色：`02_知识资产构建师`

## 1. 企业信息候选条目

```yaml
enterprise_info_candidate:
  item_id: ""
  company_profile: ""
  capabilities:
    - capability: ""
      source_id: ""
      evidence_excerpt: ""
  certifications:
    - name: ""
      scope: ""
      source_id: ""
  service_scope:
    - ""
  advantages:
    - claim: ""
      evidence: ""
      evidence_level_hint: "A/B/C/D"
      status: "fact/supported_hypothesis/needs_validation"
  team:
    - ""
  development_stage: ""
  source_ids: []
  missing_fields: []
  inference_notes: []
```

## 2. 企业信息抽取规则

- 公司事实来自公司介绍、官网、PPT、证书。
- 能力事实必须有设备、流程、团队、交付、认证或案例支撑。
- 优势主张必须绑定证据，否则标记 `needs_validation`。
- 服务范围必须具体到客户类型、行业、订单类型或市场。
- 团队经验可以记录，但不能替代企业能力证据。

## 3. 产品信息候选条目

```yaml
product_info_candidate:
  item_id: ""
  product_line: ""
  sku: ""
  features:
    - feature: ""
      source_id: ""
  parameters:
    - name: ""
      value: ""
      source_id: ""
  use_cases:
    - ""
  customization:
    - ""
  delivery:
    moq: ""
    sampling_cycle: ""
    production_cycle: ""
    logistics: ""
    after_sales: ""
  buyer_fit:
    - buyer_type: ""
      reason: ""
      status: "fact/inferred/needs_validation"
  proof:
    - type: ""
      source_id: ""
  missing_fields: []
  inference_notes: []
```

## 4. 产品信息抽取规则

- 参数是事实，卖点是解释，不能混写。
- 卖点必须对应使用场景。
- 定制能力必须尽量拆成尺寸、颜色、LOGO、包装、工艺。
- 交付能力必须尽量拆 MOQ、打样、生产周期、物流、售后。
- 后台数据只作为产品表现证据，不单独成为第五库。

## 5. 禁止

- 禁止把“品质好”“服务好”直接写成优势。
- 禁止无来源填 MOQ、交期、参数。
- 禁止把适合所有买家写进 `buyer_fit`。
