## 2. 问答抽取规则

- 保留客户原始问题。
- 可以生成归一问题，但不能删除原文。
- 必须补买家语境；没有语境则标记缺口。
- 回答候选必须说明来自已有回答、销售经验还是模型推断。
- 答案不能只是一段话，至少拆出 `core_point`；能拆则继续拆事实、案例、步骤、边界和可复用公式。
- 必须抽取 `application_scenarios`，说明这个问答能在哪个经营动作中落地。
- 异议类型必须标注。
- 必须抽取 `tags` 和 `question_keywords`，用于后续检索、内容选题、推流关键词和六组分发。

## 3. 答案结构拆分规则

`answer_structure` 的目的不是把答案写复杂，而是让答案可复用、可检索、可下放。

| 字段 | 抽取口径 |
|---|---|
| `core_point` | 这条回答最关键的判断或立场 |
| `facts` | 能支撑回答的企业事实、产品事实、数据、证书、流程 |
| `cases` | 回答里可引用的客户案例或类似经验 |
| `action_steps` | 客户或销售下一步可以怎么做 |
| `limits` | 交期、MOQ、定制、认证、价格等边界 |
| `reusable_formula` | 可迁移的话术、判断框架或处理方法，必须标明是否待验证 |

## 4. 应用场景、标签和关键词规则

```yaml
qa_retrieval_fields:
  application_scenarios:
    - "sales_reply"
    - "quote_explanation"
    - "detail_page_faq"
    - "content_topic"
    - "sales_training"
    - "private_domain_followup"
  tags:
    - "price"
    - "delivery"
    - "customization"
    - "certification"
    - "buyer_trust"
  question_keywords:
    - "MOQ"
    - "sample time"
    - "custom logo"
```

标签用于分类，关键词用于检索。二者可以重叠，但不能完全为空。

## 6. 案例抽取规则

一个正式案例候选至少要有：

1. 客户是谁。
2. 遇到什么问题。
3. 我们怎么解决。
4. 结果如何。
5. 能否公开。

缺结果时，只能是案例线索。
缺公开权限时，只能内部使用。
只有老板回忆时，标记 `needs_validation`。

## 7. 禁止

- 禁止编造客户结果。
- 禁止公开客户名或成交价。
- 禁止把销售经验直接当成已验证方法。
- 禁止没有买家语境的问答进入正式候选。
- 禁止没有具体问题、应用场景、标签和关键词的问答进入正式候选。
