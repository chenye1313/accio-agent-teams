# 经营分析有效性审计与 Skill 升级建议

版本：v0.1  
日期：2026-05-23  
对象：外贸经营定位专家 Team

## 1. 总判断

当前 Team 对客户有用，但它现在更像“经营定位咨询初稿生成器”，还不是完整的“经营分析决策系统”。

它能做好的事：

1. 把商家的产品、能力、目标买家整理清楚。
2. 逼迫商家做取舍，不再说“什么客户都做”。
3. 把卖点转成买家需求，再转成阿里页面表达。
4. 标出哪些优势有证据，哪些话暂时不能说。
5. 给出 7 天 / 30 天 / 90 天行动清单。

它现在不够的地方：

1. 店铺后台经营数据没有成为强制输入。
2. 销售数据、询盘数据、产品表现数据没有进入前置诊断。
3. 公开抓取偏“店铺表面信息”，不足以判断经营真实问题。
4. MetricQualityGate 现在偏后置质检，不能替代前置经营诊断。
5. 每个结论是否足够支撑决策，还缺一个“决策证据充分性”判断。

一句话：如果没有商家自己的经营数据，这套 Team 只能输出“有逻辑的假设版”；如果接入店铺数据、询盘数据、成交数据，它才会变成“能指导经营动作的分析系统”。

## 2. 经营分析必须接入哪些数据

经营定位不能只从卖点出发，也不能只从页面文案出发。有效路径应该是：

```text
当前经营事实
  -> 店铺流量与询盘结构
  -> 产品 / 客户 / 市场匹配度
  -> 买家真实需求
  -> 供给能力证据
  -> 经营定位
  -> 页面与询盘动作
  -> 指标验证
```

### 2.1 自有店铺后台数据

这些数据必须尽量进入 Intake 阶段：

| 数据 | 用途 |
|---|---|
| 曝光量 | 判断当前流量入口是否够 |
| 点击量 / CTR | 判断主图、标题、关键词是否吸引目标买家 |
| 访客 / PV | 判断店铺访问质量和页面深度 |
| 询盘数 | 判断页面是否能转化 |
| 询盘来源产品 | 判断哪些 SKU 真的带来商机 |
| 最常浏览产品 | 判断买家真实兴趣 |
| 最常搜索关键词 | 判断市场端用什么词找产品 |
| 订单数 / 成交金额 | 判断真正卖出去的不是哪些“想卖”的产品 |
| 国家 / 地区分布 | 判断目标市场是否与实际市场匹配 |
| 广告花费 / 平均点击成本 | 判断投放是否打到对的人 |
| 回复时效 | 判断转化问题是不是业务响应造成的 |

阿里官方 Seller Central 公开资料显示，卖家分析后台至少会涉及访客 / PV、询盘、最常浏览产品、最常搜索关键词；账号优化数据还包括曝光、点击、花费、平均点击成本、访客增长、店铺访问、询盘、订单等。

### 2.2 询盘与销售数据

后台指标只能告诉我们“发生了什么”，询盘和成交数据才能告诉我们“谁在买、为什么买、什么没成交”。

建议商家导出或整理：

| 数据 | 用途 |
|---|---|
| 最近 30-90 天询盘文本 | 判断真实买家需求、焦虑和用途 |
| 询盘买家国家 | 校准目标市场 |
| 询盘买家身份 | 区分经销商、工程商、品牌商、终端买家 |
| 采购量 / MOQ 诉求 | 判断适合小单、项目还是批发 |
| 报价后是否成交 | 判断价格、信任或交付哪个环节卡住 |
| 成交订单产品 | 判断主推产品和真实产品机会是否一致 |
| 毛利 / 客单价 | 判断定位是否有利润空间 |
| 复购客户 | 判断是否具备长期客户价值 |

### 2.3 店铺内容数据

这些数据用于判断“表达是否匹配买家”：

| 数据 | 用途 |
|---|---|
| 首页首屏截图 | 判断客户 3 秒内能否知道你是谁 |
| 产品标题 | 判断关键词是否面向正确买家 |
| 主图 / 视频 | 判断是否展示证据，而不是只展示产品 |
| 详情页模块 | 判断有没有信任资产和 CTA |
| 证书 / 案例 / 评价 | 判断差异化是否有证据 |
| 当前 Slogan | 判断是否空泛或偏离真实买家 |

### 2.4 公开市场与竞品数据

公开抓取只解决“市场和竞品参照”，不能替代自有经营数据。

建议抓取：

| 抓取对象 | 具体抓什么 | 用途 |
|---|---|---|
| 阿里搜索结果页 | 核心关键词下 Top 产品标题、主图、价格、MOQ | 判断红海表达和价格带 |
| 竞品店铺首页 | 首屏定位、类目结构、认证、年限、主推产品 | 判断可错位空间 |
| 竞品产品详情页 | 价格、起订量、规格、交付承诺、证据资产 | 判断自身优势是否真的稀缺 |
| RFQ / 采购需求 | 买家国家、采购量、用途、痛点 | 校准买家真实需求 |
| Google / 行业搜索 | 常用关键词、应用场景词、趋势 | 辅助判断海外买家语言 |

公开抓取的结论必须降级处理：只能作为市场参照，不能直接替代客户自己的成交证据。

## 3. 这些数据应该进入哪个角色

现在的机制里，数据入口主要在 `01_Intake资料采集官`，但这还不够。正确做法是：

```text
01 Intake 资料采集官
  负责收集和归一资料

新增 / 强化：经营数据诊断 Skill
  负责把后台、询盘、成交数据转成经营现状判断

02 BusinessModel
  用数据验证“到底是哪盘生意”

03 BuyerJTBD
  用询盘和成交文本验证“真实买家是谁”

04 SupplyMapping
  用证据资产验证“哪些能力能说”

05 Differentiation
  用数据和证据收束定位

06 Expression
  把定位转页面

07 MetricQualityGate
  判断上线后看哪些指标

00 Lead
  判断证据是否足够做决策
```

关键点：**经营数据必须在 BusinessModel 和 BuyerJTBD 前面进入，而不是等报告快写完时才做指标质检。**

## 4. 当前机制的优点

### 4.1 有角色分工

8 个角色的边界清楚：资料、模式、买家、能力、差异化、表达、指标、总控没有混成一个大 Agent。

### 4.2 有证据意识

SupplyMapping 会阻止没有证据的优势进入定位，这是对商家有价值的。很多商家最大的问题不是不会说，而是乱说。

### 4.3 有客户版表达规则

已经新增客户版优先和去黑话规则，能减少“内部看着专业，客户看不懂”的问题。

### 4.4 有 7/30/90 天动作

这让报告不止停留在“定位建议”，还能变成店铺、素材、询盘、指标的行动计划。

## 5. 当前机制的主要问题

### 5.1 Intake 仍然偏资料表，不够像经营诊断

现在 Intake 能收集店铺、产品、客户、证据，但对“经营现状”的结构不够强。

缺少：

- 流量漏斗。
- 询盘结构。
- 成交结构。
- SKU 表现。
- 国家市场表现。
- 投放成本。
- 回复时效。

### 5.2 BusinessModel 可能被“客户想做的生意”带偏

客户说自己想做品牌、工程、分销，不等于事实支持。BusinessModel 必须看：

- 成交订单来自哪类客户。
- 询盘来自哪类客户。
- 高毛利订单来自哪类客户。
- 复购来自哪类客户。

否则会把愿望当成经营模式。

### 5.3 BuyerJTBD 可能被公开店铺语义带偏

只看产品和店铺，会推断出“可能买家”。真正买家要从询盘文本、报价沟通、成交记录里校准。

### 5.4 公开抓取不够闭环

抓竞品页面能知道别人怎么说，但不能知道客户自己为什么没成交。公开抓取只能回答：

- 市场上大家怎么表达。
- 价格 / MOQ 大概在哪里。
- 哪些证据资产是行业标配。
- 有没有错位表达机会。

不能回答：

- 我自己的询盘为什么不准。
- 哪个产品真正赚钱。
- 哪个买家值得优先服务。
- 哪个页面改动能带来成交。

### 5.5 缺“是否足够做决策”的硬门

当前有质量门，但还需要一个更硬的判断：

```text
这个结论是否足够支撑经营决策？
如果不够，缺哪类证据？
继续输出会不会伪装确定性？
```

## 6. 建议新增 / 强化的 Skill

### P0：必须新增

#### 1. 店铺经营数据接入 Skill

建议归属：`01_Intake资料采集官`

功能：

- 接收阿里后台截图 / Excel / 手工表。
- 归一曝光、点击、访客、询盘、订单、关键词、产品表现、国家分布。
- 输出 `store_performance_snapshot.yaml`。

解决的问题：

- 让 Team 从“客户口述”升级到“经营事实”。
- 防止后续角色只基于公开店铺做推断。

核心输出：

```yaml
store_performance_snapshot:
  period:
  exposure:
  clicks:
  ctr:
  visitors:
  page_views:
  inquiries:
  orders:
  top_viewed_products:
  top_inquiry_products:
  top_searched_keywords:
  top_buyer_countries:
  ad_spend:
  avg_click_cost:
  response_time:
  data_gaps:
```

#### 2. 询盘与成交结构分析 Skill

建议归属：新增给 `03_BuyerJTBD买家任务分析师`，也可由 Intake 先清洗。

功能：

- 读取最近 30-90 天询盘文本。
- 按国家、买家身份、采购量、用途、关心点分类。
- 标记有效询盘、低质询盘、价格敏感询盘、项目型询盘、品牌型询盘。
- 输出真实买家分层。

解决的问题：

- 校准 A 类买家，不再只靠产品语义推断。
- 判断定位是否会吸引更准的客户。

核心输出：

```yaml
inquiry_buyer_segments:
  period:
  total_inquiries:
  valid_inquiry_rate:
  segments:
    - buyer_type:
      country:
      main_need:
      purchase_volume:
      anxiety:
      deal_potential:
  recommended_priority_buyer:
  mismatch_warning:
```

#### 3. SKU / 产品经营表现分析 Skill

建议归属：`02_BusinessModel生意模式分析师`

功能：

- 分析曝光、点击、询盘、成交、毛利对应到产品。
- 判断“想主推的产品”和“真实带来经营结果的产品”是否一致。
- 输出主推产品、辅助产品、该下架或弱化的产品。

解决的问题：

- 避免定位围绕老板想卖的产品，而不是客户真实买和有利润的产品。

核心输出：

```yaml
sku_performance_map:
  hero_products:
  inquiry_products:
  revenue_products:
  margin_products:
  weak_products:
  product_positioning_recommendation:
```

#### 4. 公开市场与竞品抓取 Skill

建议归属：`04_SupplyMapping供给能力映射师` 或新增给 Lead 调度。

功能：

- 抓阿里搜索结果、竞品店铺、竞品产品页、价格、MOQ、认证、页面信任资产。
- 输出竞品表达对照和市场参照。

解决的问题：

- 判断自己的卖点是不是行业通用。
- 找到真正可以错位表达的地方。

核心输出：

```yaml
market_competitor_scan:
  keywords:
  competitors:
    - name:
      positioning:
      price_band:
      moq:
      trust_assets:
      page_claims:
  market_common_claims:
  differentiation_openings:
```

#### 5. 决策证据充分性 Skill

建议归属：`00_Lead经营定位总控`

功能：

- 每个核心决策前检查证据是否足够。
- 判断输出是“可决策”“可讨论”“仅假设”“禁止下结论”。
- 明确缺哪类数据。

解决的问题：

- 防止报告看起来很完整，但不足以指导经营决策。

核心输出：

```yaml
decision_sufficiency_gate:
  decision:
  evidence_used:
    internal_store_data:
    inquiry_data:
    sales_data:
    competitor_data:
    capability_proof:
  sufficiency_level: "decision_ready | discussion_ready | hypothesis_only | blocked"
  missing_evidence:
  allowed_output:
```

### P1：下一轮增强

#### 6. 页面漏斗审计 Skill

归属：`06_Expression表达落地师`

功能：

- 对照当前首页、产品页、详情页，判断每一屏是否承接目标买家。
- 输出页面改版优先级。

#### 7. 投放关键词与定位匹配 Skill

归属：`07_MetricQualityGate经营罗盘质检官`

功能：

- 把后台最常搜索关键词、广告关键词和定位词做匹配。
- 判断投放吸引的是不是目标买家。

#### 8. 30 天复盘 Skill

归属：`07_MetricQualityGate经营罗盘质检官`

功能：

- 上线 30 天后读取指标变化。
- 判断是定位问题、页面问题、投放问题、产品问题还是业务响应问题。

## 7. 角色级调优建议

| 角色 | 当前够不够 | 必须调优 |
|---|---|---|
| Intake | 不够 | 增加店铺后台数据、询盘、成交、SKU 经营字段 |
| BusinessModel | 中等 | 用 SKU / 成交 / 毛利数据验证生意模式 |
| BuyerJTBD | 中等 | 用询盘文本和成交客户校准真实买家 |
| SupplyMapping | 中等 | 增加竞品抓取和行业证据对照 |
| Differentiation | 较好 | 增加“数据支持的定位”判断，不只看能力证据 |
| Expression | 较好 | 增加当前页面漏斗审计，不只生成新 7 屏 |
| MetricQualityGate | 不够前置 | 从后置质检升级为前后两端：上线前基线 + 上线后复盘 |
| Lead | 需要强化 | 增加决策证据充分性门，不足就降级输出 |

## 8. 新的推荐运行流程

建议从现在的线性流程升级为“数据诊断 + 定位生成 + 经营复盘”三段：

```text
第一段：经营现状诊断
  店铺数据接入
  询盘 / 成交结构分析
  SKU / 国家 / 关键词表现分析

第二段：经营定位生成
  生意模式判断
  买家任务分析
  能力证据映射
  差异化定位
  页面表达

第三段：经营验证复盘
  绑定指标
  上线 7/30/90 天看数据
  判断哪里需要返工
```

## 9. 判断报告是否真的有用的标准

一份报告只有满足下面 6 条，才算对客户真的有用：

1. 能说清当前经营问题，而不是只写未来定位。
2. 能说明哪些结论来自后台数据，哪些只是推断。
3. 能指出最值得先服务的客户，并说明为什么。
4. 能告诉客户哪些页面和话术马上要改。
5. 能告诉客户哪些证据必须补，不补不能上线。
6. 能绑定 30 天和 90 天的经营指标，用数据回头验证。

如果缺第 1 条和第 2 条，这份报告就是“咨询感强，但经营决策弱”。

## 10. 立即行动建议

### 先不急着加新角色

当前 8 个角色结构还能用。先补 Skill 和输入字段，不急着拆成 10 个 Agent。

### 先新增 5 个 P0 Skill

1. `store-performance-intake`：店铺经营数据接入。
2. `inquiry-sales-segmenter`：询盘与成交结构分析。
3. `sku-performance-mapper`：SKU / 产品经营表现分析。
4. `market-competitor-scanner`：公开市场与竞品抓取。
5. `decision-sufficiency-gate`：决策证据充分性检查。

### 再改 3 个共享知识

1. `02_输入字段与完整度.md`：增加经营数据字段。
2. `03_方法路由卡.md`：把数据诊断放到生意模式前。
3. `05_经营罗盘指标映射.md`：增加上线前基线和上线后复盘。

### 最后做一个真实客户回放

用 Sweet Home 或另一个真实店铺，按新流程跑一次：

```text
后台数据 + 询盘数据 + 公开竞品
  -> 经营定位
  -> 页面改版建议
  -> 30 天指标复盘计划
```

如果没有后台数据，只能输出假设版，并明确告诉客户：这不是最终经营决策报告。
