# 04 案例回放验收规则

版本：v0.1  
适用角色：`07_MetricQualityGate经营罗盘质检官`

## 1. 用途

每次 Team 输出后，必须用案例回放方式验证：

```text
输入是否被正确使用？
关键判断是否可追溯？
输出是否覆盖能力问题？
指标是否绑定？
```

## 2. Case 2 HOTBLIZZARD 回放基准

来源：`tests/case2_hotblizzard_replay.md`

### 2.1 应通过项

| 检查项 | 最低要求 |
|---|---|
| 生意模式 | 能识别品牌孵化供应链伙伴，不只是 slogan |
| A 类买家 | 街头潮牌创业者 / 小型潮流服饰品牌主 |
| JTBD | 从买产品升级为低库存风险验证新品并放量 |
| 差异化 | 小单测试、趋势定制、可信供应链证据 |
| 信任资产 | 认证、产能、合作品牌，同时标注案例缺口 |
| 7 屏表达 | 可输出 7 屏 brief |
| 经营指标 | 绑定有效询盘率、CTR、访客转化、报价订单率 |

### 2.2 可接受缺口

```yaml
acceptable_gaps:
  - "公开案例细节缺失"
  - "真实询盘和后台数据缺失"
  - "合作品牌展示授权未确认"
```

可接受缺口必须写进 `revision_requests`，不能假装完整。

## 3. Case 1 Sweet Home 回放基准

来源：`process/cases/case1_sweethome_extract.md`

最低要求：

- 能解释为什么商用客户优先。
- 能解释一站式户外空间方案如何对应买家 Job。
- 能解释 CAD/3D、安装、本地服务、120% 配件为什么是信任资产。
- 能生成与原案例 7 屏结构相近的 brief。
