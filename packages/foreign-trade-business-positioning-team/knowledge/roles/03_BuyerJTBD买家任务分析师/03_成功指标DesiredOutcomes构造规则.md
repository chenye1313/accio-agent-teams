# 03 成功指标 Desired Outcomes 构造规则

版本：v0.1  
适用角色：`03_BuyerJTBD买家任务分析师`

## 1. 核心判断

痛点不是成功指标。

BuyerJTBD 必须把“客户很痛”改写成：

```text
客户如何衡量任务完成得更好？
```

成功指标要能被后续 SupplyMapping 拿来匹配企业能力。

## 2. Desired Outcome 句式

推荐句式：

```text
Minimize / Reduce / Increase / Improve + 衡量对象 + in/when/while + 具体情境
```

中文表达：

```text
在[情境]中，降低/提升/缩短/增加[可衡量结果]。
```

## 3. 不合格指标

| 错误写法 | 问题 |
|---|---|
| 产品质量好 | 不是买家的衡量方式 |
| 服务好 | 太泛 |
| 价格低 | 只是一种可能指标 |
| 交期快 | 缺少情境 |
| 提高品牌 | 无法判断交付是否成功 |

## 4. 合格指标示例

### 4.1 品牌孵化供应链型

| 买家任务 | 成功指标 |
|---|---|
| 低风险验证新品 | 缩短从设计概念到可售样衣的周期 |
| 控制首批库存风险 | 降低首批备货压错款的概率 |
| 判断是否放量 | 提高小单测试反馈对量产决策的可信度 |
| 快速上新 | 缩短趋势出现到商品上架的时间 |
| 形成品牌风格 | 提高款式、工艺、标签、包装的一致性 |

### 4.2 项目解决方案型

| 买家任务 | 成功指标 |
|---|---|
| 降低项目协调成本 | 减少设计、生产、安装之间的沟通次数 |
| 控制总成本 | 降低项目超预算概率 |
| 减少返工 | 提高方案一次确认率 |
| 保障售后 | 缩短维修配件响应周期 |
| 证明可靠性 | 增加可复用的相似项目案例 |

### 4.3 工厂供货 / OEM 型

| 买家任务 | 成功指标 |
|---|---|
| 稳定补货 | 降低断货或延迟交付概率 |
| 控制合规风险 | 提高认证、检测、质检一次通过率 |
| 提高毛利 | 降低单位采购综合成本 |
| 新品开发 | 缩短打样到确认周期 |

## 5. 成功指标分类

至少输出三类：

| 类型 | 例子 |
|---|---|
| 时间指标 | 缩短打样周期、缩短上架周期 |
| 风险指标 | 降低库存风险、降低项目返工风险 |
| 成本指标 | 降低协调成本、降低综合采购成本 |
| 收益指标 | 提高爆款命中率、提高复购概率 |
| 信任指标 | 提高样品确认率、提高案例可复用性 |

## 6. 与 SupplyMapping 的交接

每个成功指标必须交给 SupplyMapping 去匹配企业能力：

```yaml
desired_outcome:
  statement: ""
  outcome_type: "time | risk | cost | revenue | trust"
  buyer_role: ""
  linked_capability_needed:
    - ""
```

如果一个指标找不到任何企业能力支撑，应标为 `unsupported_desired_outcome`。
